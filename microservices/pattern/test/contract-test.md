# Contract Testing

**Contract testing** verifies that **two services agree on the API contract** (request + response) **without deploying both services together**.

The contract specifies:

* Endpoint
* HTTP method
* Request body
* Headers
* Response status
* Response body

Instead of running an end-to-end test, both sides independently verify the same contract.

---

## Example

Suppose we have:

```
Frontend (Consumer)
        |
        v
User Service (Provider)
```

The frontend calls:

```http
GET /users/1
```

Expected response:

```json
{
  "id": 1,
  "name": "John"
}
```

This expectation is the **contract**.

---

# Consumer-side Contract Test

The **consumer** writes the contract.

Example (TypeScript + Pact):

```ts
// consumer.test.ts

provider
  .given("User 1 exists")
  .uponReceiving("Get user")
  .withRequest({
    method: "GET",
    path: "/users/1",
  })
  .willRespondWith({
    status: 200,
    body: {
      id: 1,
      name: "John",
    },
  });

// Frontend code runs against Pact mock server
const user = await api.getUser(1);

expect(user.name).toBe("John");
```

This generates a contract file:

```json
{
  "request": {
    "method": "GET",
    "path": "/users/1"
  },
  "response": {
    "status": 200,
    "body": {
      "id": 1,
      "name": "John"
    }
  }
}
```

No real backend is needed.

---

# Provider-side Contract Test

The backend loads the contract generated by the consumer.

```ts
describe("Provider Verification", () => {

    verifyContract(contractFile);

});
```

Backend actually serves:

```http
GET /users/1
```

```json
{
    "id": 1,
    "name": "John"
}
```

If the response matches the contract:

```
✔ Contract Passed
```

If backend returns

```json
{
    "id": 1,
    "fullName": "John"
}
```

it fails because

```
Expected:
name

Received:
fullName
```

The provider broke the contract.

# Consumer-Driven Contract (CDC)

The **consumer defines what it needs**.

```
Consumer
    |
    | generates contract
    v
Contract File
    |
    v
Provider verifies it
```

The provider only promises to satisfy what consumers require.

Example:

Frontend only needs

```json
{
    "name": "John"
}
```

Contract:

```json
{
    "name": "John"
}
```

Backend may actually return

```json
{
    "id": 1,
    "name": "John",
    "age": 35,
    "address": "...",
    "phone": "..."
}
```

Still passes because the consumer didn't care about the extra fields.

This is **Consumer-Driven Contract Testing**.

---

# Why Consumer-Driven?

Without CDC:

Backend changes

```json
{
    "name": "John"
}
```

to

```json
{
    "fullName": "John"
}
```

Everything compiles.

Backend unit tests pass.

But frontend breaks in production.

CDC catches this **before deployment**.

# Workflow

```
Consumer
    |
    | 1. Run contract test
    |
    v
Contract File
    |
    | publish
    v
Contract Broker
    |
    | provider downloads
    v
Provider Verification
```

If verification succeeds:

```
Safe to deploy
```

---

# Consumer-side vs Provider-side

| Consumer-side           | Provider-side            |
| ----------------------- | ------------------------ |
| Written by API consumer | Written by API provider  |
| Uses mock server        | Uses real implementation |
| Generates contract      | Verifies contract        |
| Doesn't need backend    | Doesn't need frontend    |
| Defines expectations    | Confirms implementation  |

---

# Consumer-Driven Contract vs Traditional Integration Test

| Integration Test              | Contract Test                        |
| ----------------------------- | ------------------------------------ |
| Real consumer + real provider | One side at a time                   |
| Slow                          | Fast                                 |
| Needs deployed services       | No deployment required               |
| Detects runtime issues        | Detects API compatibility issues     |
| Hard to debug                 | Easy to pinpoint contract violations |

---

## In one sentence

* **Contract testing:** Verifies that a consumer and provider agree on an API contract.
* **Consumer-side contract test:** The consumer tests against a mock provider and generates the contract.
* **Provider-side contract test:** The provider verifies its real implementation satisfies that contract.
* **Consumer-Driven Contract (CDC):** The consumer defines the contract, and the provider must honor it.
